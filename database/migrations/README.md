# Migrations Policy

Этот каталог является канонической историей схемы БД для проекта.

Правила:
- Не переименовывать уже существующие migration-файлы.
- Не исправлять "странные" имена старых файлов только ради красоты.
- Не удалять пустые или дублирующие по смыслу миграции, если у них уже есть своя версия.

Почему так:
- Goose хранит применённые версии в `goose_db_version`.
- Если файл уже был применён на стенде или в проде, его переименование или удаление ломает воспроизводимость истории.
- Поэтому файлы вроде `.goose.sql`, `.sql.sql` или с длинным числовым префиксом остаются как часть реальной истории проекта.

Как добавлять новые миграции:
- Использовать новый уникальный version prefix.
- Писать миграции в формате Goose: `-- +goose Up` / `-- +goose Down`.
- Не редактировать старые применённые миграции, если это не абсолютно необходимо.

Что считается нормализацией:
- Все реальные migration-файлы должны лежать в git.
- История в каталоге должна совпадать с тем, что реально поднималось на стендах.
